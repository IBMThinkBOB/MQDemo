Lab Overview

What you are doing: In this lab you will connect to a dedicated LinuxONE VM, download a task specification file, and feed it into an AI agent (Bob). The agent will execute every step and produce the following 4 shell scripts as its output:

  • start-mq-demo.sh — starts IBM MQ and confirms it is running
  • stop-mq-demo.sh — gracefully stops IBM MQ
  • cleanup-mq-demo.sh — removes all queue manager data for a clean reset
  • check-build-status.sh — shows the full state of your MQ environment at a glance

Your job is not to run the commands yourself. Your job is to connect to your VM, download the spec, give it to Bob, and verify the 4 scripts appear with the correct output. Bob does the rest.

Time Required: 20–30 minutes
Difficulty: Beginner-friendly
Prerequisites: SSH access to your VM and access to Bob

Step 1

Connect to Your LinuxONE VM ~3 min

What you are doing in this step: Opening a terminal connection to your remote LinuxONE server. This is the only technical prerequisite before handing off to Bob.

Your VM is a Linux server running on IBM LinuxONE hardware. You access it over the internet using SSH — a secure, encrypted terminal connection. Think of it as opening a command prompt that runs directly on your remote server.

1.1 Claim Your SSH Key

The keys are titled oma# — please use the key information you were handed. Navigate to the URL below in your browser, replacing # with your key number:

https://marist.ddns.net/sshkeys/oma#.pem

1.2 Set Your Key Permissions

Before connecting, set the correct permissions on your key file — SSH will refuse to use a key that is too open:

chmod 600 /path/to/your-key-file.pem

If you are on Windows you may have to use .ppk. If .pem doesn't work, please try that.

1.3 Connect via SSH

Now use your key file to connect. Replace the placeholders with the values from your secure connection details from the information you were handed:

ssh -i /path/to/your-key-file.pem linux1@your-server-ip

Example of what a real command looks like:

ssh -i ~/.ssh/linuxone-lab.pem linux1@192.168.1.100

The -i flag tells SSH which key file to use. linux1 is the username. 192.168.1.100 is the server address. Yours will differ.

1.4 Accept the Host Fingerprint

On your very first connection, SSH does not yet know this server. It will ask you to confirm its identity:

The authenticity of host '192.168.1.100' can't be established.
Are you sure you want to continue connecting (yes/no)?

Type yes and press Enter. SSH will save the server's fingerprint so it does not ask again.

You are connected when you see a prompt like this:
[username@hostname ~]$
Everything you type from now on runs on the LinuxONE VM, not your laptop.

Final Checklist

Troubleshooting

Cannot connect via SSH

  • Run chmod 600 your-key-file.pem — SSH rejects keys with loose permissions
  • Double-check the server IP and username from your connection details email
  • If your organisation requires a VPN, confirm it is connected before trying SSH
  • Contact event support with the exact error message SSH printed

Download is very slow or fails

  • Check internet access from the VM: ping -c 3 ibm.com
  • If ping fails, ask your facilitator — the VM may need a proxy configured
  • Ask your facilitator if a pre-staged copy of the installer is available on the VM

RPM installation fails

  • Confirm you accepted the license first: cd ~/mq-install/MQServer && sudo ./mqlicense.sh -accept
  • Check you have disk space: df -h ~ (need at least 2 GB free)
  • Confirm sudo works: sudo -v
  • Paste the exact error line to event support

Queue manager QM1 will not start

  • Check the MQ error log for the specific reason: sudo tail -50 /var/mqm/qmgrs/QM1/errors/AMQERR01.LOG
  • Verify the mqm user exists: id mqm
  • Verify permissions on the data directory: ls -la /var/mqm — should be owned by mqm
  • If QM1 was never created, run: sudo -u mqm crtmqm QM1

amqsput or amqsget fails

  • Confirm QM1 is running: sudo -u mqm dspmq
  • Confirm EVENT.Q1 exists: sudo -u mqm bash -c "echo 'DISPLAY QLOCAL(EVENT.Q1)' | runmqsc QM1"
  • If the queue is missing, recreate it: sudo -u mqm bash -c "echo 'DEFINE QLOCAL(EVENT.Q1) REPLACE' | runmqsc QM1"

A script gives "Permission denied" when you run it

  • The script is not executable. Fix it: chmod +x ~/the-script-name.sh

Quick Reference

Your 4 Scripts

# Start MQ
~/start-mq-demo.sh

# Stop MQ
~/stop-mq-demo.sh

# Check status
~/check-build-status.sh

# Full cleanup (destructive)
~/cleanup-mq-demo.sh

Queue Manager Commands

# Check status
sudo -u mqm dspmq

# Start queue manager
sudo -u mqm strmqm QM1

# Stop queue manager
sudo -u mqm endmqm QM1

Messaging Commands

# Send a message
echo "text" | sudo -u mqm \
  /opt/mqm/samp/bin/amqsput EVENT.Q1 QM1

# Receive a message (destructive)
sudo -u mqm \
  /opt/mqm/samp/bin/amqsget EVENT.Q1 QM1

# Browse messages (non-destructive)
sudo -u mqm \
  /opt/mqm/samp/bin/amqsbcg EVENT.Q1 QM1

Queue Operations

# Create a queue
echo 'DEFINE QLOCAL(EVENT.Q1) REPLACE' \
  | sudo -u mqm runmqsc QM1

# Check queue depth
echo 'DISPLAY QLOCAL(EVENT.Q1) CURDEPTH' \
  | sudo -u mqm runmqsc QM1

# List all queues
echo 'DISPLAY QLOCAL(*)' \
  | sudo -u mqm runmqsc QM1

What You've Learned

Core Concepts

  • Queue Manager (QM1) — the MQ service that owns queues and routes messages between applications
  • Queue (EVENT.Q1) — a named storage location for messages; like a mailbox that holds messages until they are collected
  • PUT — placing a message on a queue (amqsput); the sender does not need to know who will read it or when
  • GET — retrieving and removing a message from a queue (amqsget); the message is gone after this
  • BROWSE — reading a message without removing it (amqsbcg); useful for debugging and monitoring
  • FIFO — messages come out in the same order they went in: First In, First Out
  • Persistence — messages are written to disk inside /var/mqm and survive a restart

Real-World Applications

The pattern you practised — put a message, queue holds it, get the message — is used in production at scale:

  • Banking: Transaction events are queued so no payment is lost even if a downstream system is temporarily unavailable
  • Healthcare: Patient records are exchanged between hospitals via queues that guarantee delivery
  • E-commerce: Order confirmations, inventory updates, and shipping notifications all flow through message queues
  • Microservices: Independent services communicate asynchronously through queues instead of calling each other directly
